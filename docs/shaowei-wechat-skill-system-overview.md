# 少威公众号 Skill 体系沉淀说明（对外分享版）

## 一、这套体系是怎么来的

少威公众号 skill 体系，并不是一开始就设计出来的一套“大而全工具”。

它是从真实写作动作里，一步步拆职责、固格式、做闭环，最后沉淀出来的一套可复用、可迁移的公众号生产体系。

它的形成顺序大致是：

1. 先有稳定文风
2. 再有稳定版式
3. 再从纯文字升级到图文一体
4. 最后形成发布出口与总控闭环

所以，这套体系的核心价值，不在于功能数量，而在于：

- 每一层都来自真实 SOP
- 每一层职责边界清晰
- 既能单独复用，也能整套编排
- 不依赖单次手工操作，而是可沉淀、可迁移

---

## 二、为什么要拆成多个 skill，而不是做成一个大黑盒

因为公众号生产并不是一个动作，而是一组不同层次的问题。

如果把它们混在一起，会导致：
- 内容改写和版式渲染耦合
- 配图判断和生图执行耦合
- 有图和图文交付耦合
- 迁移时无法局部替换

因此，这套体系采用的是“原子能力层 + 流程编排层 + 套件入口层”的分层设计。

### 1. 原子能力层
负责单一职责，便于单独迁移和复用。

包括：
- `shaowei-wechat-article`
- `shaowei-wechat-writer`
- `shaowei-wechat-illustrator`
- `shaowei-wechat-image-generator`
- `wechat-article-publisher`

### 2. 流程编排层
负责把原子能力串成闭环。

包括：
- `shaowei-wechat-image-pipeline`
- `shaowei-wechat-pipeline`

### 3. 套件入口层
负责总说明、迁移说明、组合关系和推荐用法。

包括：
- `shaowei-wechat-suite`

这种设计的好处是：
- 可以单独迁移某个能力
- 可以只迁一段子流程
- 也可以整套迁移到其他 agent 或软件中

---

## 三、每一个 skill 是如何沉淀出来的

### 1. `shaowei-wechat-article`：内容生成层

这个 skill 的出现，来自一个非常真实的问题：
用户给的往往不是一篇完整文章，而是初稿、提纲、问答、碎片观点、会议记录，甚至只是一个想法。

因此，需要先有一层专门负责：
- 整理结构
- 补齐逻辑
- 完成转场
- 形成可读的公众号正文中间稿

它沉淀的不是排版能力，而是少威稳定的内容表达方法论，包括：
- 从生活切口切入
- 把技术问题翻译成普通人能懂的话
- 用“生活切口 → 技术拆解 → 抽象升维 → 成长/选择落点”的结构组织内容
- 保持理性但有温度的叙述方式

它解决的是“文章怎么写顺”，而不是“文章怎么发布”。

---

### 2. `shaowei-wechat-writer`：固定格式渲染层 / 定稿层

这个 skill 的出现，来自另一个关键认知：
“按我的格式”不是抽象偏好，而是已经有真实样文基准的固定成品样式。

也就是说，内容写顺只是第一步。
真正的公众号成品，还需要稳定的版式规则，例如：
- 标题下的浅黄编号框
- 短段落节奏
- 一级标题红标块
- 重点句高亮
- 结尾长分隔线 + 居中 `End`
- 固定尾注分行

因此，这一层被单独拆出来，专门负责：
- 固定格式渲染
- 文章定稿
- 重点句与结构节奏强化
- 尾注和结尾结构统一

它沉淀的是少威的公众号成品标准，而不是内容生成逻辑。

---

### 3. `shaowei-wechat-illustrator`：配图规划层

当公众号从纯文字走向图文一体时，首先暴露出来的问题不是“怎么生图”，而是“该不该加图、哪里加图、每张图承担什么作用”。

因此，需要先有一层专门负责：
- 判断文章是否适合配图
- 判断适合 1 张头图还是多张文中配图
- 标注图片插入位置
- 给出每张图的作用解释
- 协调图文节奏与图注建议

它沉淀的是配图判断逻辑，而不是生图技术。

这意味着，图片不再只是装饰，而是文章节奏和理解路径的一部分。

---

### 4. `shaowei-wechat-image-generator`：生图执行层

当配图需求明确后，接下来的问题是：
不是“给几个 prompt 建议”，而是“把图真的生出来”。

因此，这一层被独立出来，专门负责：
- 将图片需求转为高质量 prompt
- 选择合适尺寸、构图与风格
- 调用 OpenAI-compatible 图片接口
- 保存 request / response 文件
- 将 base64 解码为本地 png

它沉淀的不只是生图 prompt 写法，还包括公众号语境下的视觉偏好，例如：
- 解释型文章优先考虑中文白板教授图
- 图中要有中文、箭头、框、关系线
- 避免廉价 AI 感、海报风、纯氛围图误替代讲解图

这一步使“图片能力”从灵感建议变成真实执行能力。

---

### 5. `shaowei-wechat-image-pipeline`：配图闭环总控层

图片生出来，并不代表图文闭环完成。
真正的交付还需要继续完成：
- 上传微信公众号素材库
- 拿到素材 URL
- 回填正文图片链接
- 设置图注样式
- 重新发草稿箱

因此，这一层被独立出来，专门负责把：
“配图规划 → 生图执行 → 微信上传 → 正文回填 → 草稿重发”
串成一条完整链路。

它沉淀的是图文一体的交付 SOP。

这一步很重要，因为它把“有图”升级成“图真的进入公众号成品”。

---

### 6. `wechat-article-publisher`：微信发布层

前面的内容、版式、图片都准备好之后，还需要一个稳定的发布出口。

这一层负责：
- dry-run 预览
- 创建草稿
- 可选提交发布
- 返回 `draft_media_id`、`publish_id`、发布状态等结果

对少威公众号场景来说，它还承担一个边界控制作用：
默认只创建草稿，不直接正式发布。

这意味着自动化只负责把文章送到可审阅状态，而最终是否上线，保留人工决策。

因此，它沉淀的是发布边界与草稿交付出口。

---

### 7. `shaowei-wechat-pipeline`：全文闭环总控层

当内容、定稿、配图、发布都已稳定后，就不应该每次手动拼接整个流程。

因此，最终需要一个总控层，负责：
- 接收初稿 / 提纲 / 碎片想法 / 链接
- 先生成内容中间稿
- 再渲染为固定格式成品
- 当用户要求图文一体时接入配图闭环
- 最后执行 dry-run 检查并推送草稿箱

它沉淀的是少威完整的公众号工作流，而不是某一个局部能力。

它的存在，意味着这套体系已经不只是工具集合，而是一个可调用的标准 SOP。

---

### 8. `shaowei-wechat-suite`：总入口 / 架构说明层

当一套能力开始面向“迁移”和“复用”时，仅仅能运行还不够。
还必须说明：
- 每个 skill 的职责边界
- 哪些适合单独迁移
- 哪些适合打包迁移
- 推荐调用顺序是什么
- 哪些是平台无关层，哪些是平台绑定层

因此，`suite` 被沉淀出来，作为整套体系的总入口与说明书。

它本身不直接写文章，也不直接发布，但它对于迁移和扩展非常关键。

---

## 四、这套体系的分层价值是什么

这套 skill 体系真正有价值的地方，不是“会写、会排、会生图、会发布”，而是它把这些能力拆成了清晰层次。

### 第一，文风和版式被正式分层
- 内容生成层解决“写什么、怎么讲顺”
- 定稿渲染层解决“怎么变成少威的公众号成品”

### 第二，配图判断和生图执行被正式分层
- 配图规划层先决定图的角色和位置
- 生图执行层再把图真的做出来

### 第三，生图和交付闭环被正式分层
- 生图执行层负责生成文件
- 配图闭环层负责把文件送进微信正文和草稿箱

### 第四，局部能力和总控流程被正式分层
- 原子能力层便于复用和替换
- pipeline 层负责定义标准顺序
- suite 层负责帮助迁移和扩展

这种分层结构，使得体系具备四个特点：
- 可维护
- 可扩展
- 可迁移
- 可复用

## 五、为什么这套体系适合对外分享

因为它不只是一个“我会写公众号”的经验总结，而是一个典型的“个人 SOP 如何被沉淀为 AI skill 体系”的案例。

它具备很强的分享价值，原因有三点：

### 1. 它展示了如何把隐性经验变成显性规则
少威原本的公众号能力，最初是个人表达习惯。
但后来逐渐被抽象成：
- 内容规则
- 版式规则
- 配图规则
- 发布规则
- 编排规则

这说明个人风格是可以工程化的。

### 2. 它展示了如何从单次任务走向长期复用
不是每次临时写、临时排、临时发，
而是逐步形成稳定结构，并能不断复用。

### 3. 它展示了如何从“能力点”走向“闭环交付”
很多系统只有写作能力，或者只有生图能力。
但真正成熟的体系，要能交付到最后一公里，比如微信公众号草稿箱。

---

## 六、一句话总结这套体系

少威公众号 skill 体系的本质，不是做了几个工具，
而是把“少威自己的公众号 SOP”从隐性经验，变成了可调用、可迁移、可复用的分层生产系统。

如果进一步压缩成一句更适合分享的话，可以说：

> 我做的不是一个公众号写作工具，而是把自己的公众号工作流拆成了内容、版式、配图、发布四类能力，再通过 pipeline 把它们编排成可长期复用的 skill 体系。
